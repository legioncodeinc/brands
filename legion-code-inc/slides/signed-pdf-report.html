Two issues will block a Stripe risk review. Your webhook handler writes raw card numbers to console output, and a Stripe live secret is still readable in git history from April. Both are fixable in under an hour. Five other issues need attention this sprint; three more can wait.

Critical 2 findings · fix these first
Critical
Primary account number found in plain text
src/handlers/webhook.ts:142
PCI-DSS 3.4
142 │console.log('Processing card:', cardNumber)
Why it matters

Your webhook handler logs the raw card number to console output. Any service reading those logs — your hosting provider, your log aggregator, anyone with read access to the server — sees the PAN in plain text. This is a PCI-DSS 3.4 violation and the most common reason Stripe risk teams suspend accounts.

How to fix

Remove the console.log, or mask the value. The masked form should show only the last four digits: Processing card: ****1234. Rotate any logs already written; treat them as compromised.

The key was added in commit a91f2b3 on 2026-04-22 and is still readable in git history even though .env.local is now in .gitignore. Anyone with read access to the repo can pull the key and charge against your Stripe account.

How to fix

Warning
API allows requests from any origin
src/server.ts:47
OWASP A05
47 │app.use(cors({ origin: "*" }))
Why it matters

Setting CORS origin to "*" lets any website call your API and read the response. If your API uses cookie authentication, a malicious site can act as a logged-in user.

How to fix

Replace "*" with the explicit list of origins your frontend is served from: ["https://app.acme.com"]. For multiple environments, use an array.
